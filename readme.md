# アカウント認証型APIサーバーの実装解説

---

## 概要
RESTfulの設計原則に基づき、ユーザーアカウントの作成・取得・更新・削除を行うAPIサーバーです。

## 技術スタック
*   **言語**: Python 3.11
*   **フレームワーク**: FastAPI
*   **データベース**: SQLAlchemy (SQLite)
    *   *※テスト環境でのポータビリティを考慮しSQLiteを採用しています。
*   **パッケージ管理**: uv
*   **環境**: Docker

## 実装のポイント

### 1. 入力バリデーションの徹底
*   **Pydantic と 正規表現 (`re`)**: ユーザーID（英数字6〜20文字）やパスワード（英数字および指定記号8〜20文字）の文字種・文字数制限を厳密にチェックしています。
*   **制御文字の排除**: ニックネームやコメント等の文字列に制御文字が含まれていないか、`.isprintable()` メソッドを用いて検証しています。

### 2. エラーレスポンスのフォーマット統一
*   FastAPIのデフォルトのバリデーションエラー（422 Unprocessable Entity）をそのまま返さず、`@app.exception_handler` を用いてカスタム例外ハンドラを実装しました。
*   これにより、要件定義にある `{"message": "...", "cause": "..."}` という固有のJSONフォーマットと適切なHTTPステータスコード（400, 401, 403, 404など）を正確に返却しています。

### 3. Basic認証（Authorizationヘッダー）の処理フロー
本実装での認証（`get_current_user` 関数）は以下のフローで行われます。

1.  **ヘッダーの取得**: リクエストから `Authorization: Basic <Base64文字列>` を受け取ります。
2.  **デコード**: Base64文字列をサーバー側でデコードし、`user_id:password` の形式に変換・分割します。
3.  **データベース照合**: 抽出した `user_id` をキーにしてデータベースを検索し、ユーザーが存在するか確認します。
4.  **パスワード検証**: データベースに保存されているパスワードと、デコードして得られたパスワードが一致するかを検証します。
5.  **結果**: 全て一致した場合のみ、後続の処理（ユーザー情報の取得や更新など）を許可し、それ以外は `401 Authentication failed` を返却します。
